Executive Summary

Workday's Self-Serve Agent — now branded as the Sana Self-Service Agent following the $1.1B acquisition of Sana Labs (completed November 2025) — represents Workday's most significant architectural shift in a decade. Rather than building an AI layer on top of its existing monolithic HCM/Finance platform, Workday has grafted an entirely new AI-native front end (Sana) onto its deterministic back end.

The agent supports 300+ skills across HR and finance domains, connects to multiple LLM providers (Claude, OpenAI, Gemini), and is governed by the Workday Agent System of Record (ASOR) — a first-of-its-kind registry that manages AI agents alongside human employees.7

The architecture uses Model Context Protocol (MCP) and Agent-to-Agent Protocol (A2A) for interoperability, and the Flowise Agent Builder (acquired August 2025) enables customers to create custom agents via low-code tooling.5

Key differentiators: Workday's unmatched HR/Finance dataset (1 trillion+ transactions/year), the ASOR governance model, and multi-LLM flexibility. Key risks: the Sana integration is only four months old, many features are still in early access, and the competitive field is moving fast.

Confirmed Architecture Details

Workday Core: The deterministic transaction engine processes 1 trillion+ transactions per year. The Object Management Server (OMS) enforces tenant isolation. Business processes have defined start/end states — "must get done right every time" (e.g., payroll).14

Sana Agentic Layer: Supports multiple LLM providers — Claude (Anthropic), OpenAI, and Google Gemini — from a single interface. The platform includes 300+ pre-built skills for HR/finance and 400+ sub-agents built in Galileo (Sana's workflow development tool). Provides read/write access to external systems including Salesforce, Slack, Gmail, Jira, Notion, and SharePoint.1417

Data Cloud: Developed in partnership with Databricks, Snowflake, and Salesforce. Provides direct, real-time, zero-copy access to HR and finance data with bidirectional flow.21

Inferred — Not Officially Confirmed

Agentic Loop: Workday describes agents that "reason through situations, plan toward a goal, take autonomous action, and learn from experience" — textbook agentic AI implying a ReAct-style or plan-and-execute loop.

Skill Granularity & Customization

Each agent contains multiple skills manageable at a granular level within ASOR. Josh Bersin reports 400+ "sub-agents" built in Galileo specifically for HR workflows, using drag-and-drop "steps" or "prompt paths."14

  • Out-of-the-box: Pre-built skills for common HR/finance workflows
  • Customizable: Customer-configurable skills for specific business processes
  • Custom-built: Entirely new agents via Flowise Agent Builder (H1 2026, requires Extend Professional)

Skill Chaining

The system can decompose complex objectives into sequential tasks across multiple platforms. Example from Workday's marketing: "Set up a monthly workflow to review my email inbox for receipts, check them against policy, and send me a report to approve before submitting."18

Confirmed Capabilities

Per Workday's official documentation, Sana agents can:8

  1. Understand details across contexts — multi-turn conversation with context retention
  2. Reason through situations — assess conditions and determine appropriate actions
  3. Plan toward a goal — break complex requests into steps
  4. Take autonomous action within human-defined parameters — execute tasks with guardrails
  5. Learn from experience — improve over time (mechanism unspecified)

Execution Model

Users interact through a conversational dashboard that replaces traditional Workday menu navigation. The agent interprets intent, determines which skills/sub-agents are needed, checks permissions against the user's security profile, executes actions against Workday and connected systems, and returns results or requests human approval for sensitive actions.18

Human-in-the-Loop

Workday emphasizes "appropriate human oversight." The system augments rather than replaces human workers. Sensitive actions (payroll changes, financial approvals) likely require human confirmation, though the exact threshold/configuration model is not publicly documented.8

Multi-System Orchestration

Via Sana Enterprise, agents coordinate actions across Workday core (HR, Finance, Planning), Gmail/Google Drive, Microsoft Outlook/SharePoint, Slack, Jira, Notion, Salesforce, and other connected applications.14

Tenant Isolation

Workday's multi-tenant architecture uses the Object Management Server (OMS) to enforce tenant boundaries. Automatic tenancy filters ensure data requests return only tenant-specific data. AI agents inherit the same isolation — no cross-tenant data access.12

Permission Model

AI agents follow the same access rules as employees. Each agent only sees and does what it's authorized to handle. Security is managed through Workday's existing roles, security groups, and business process configurations. Agents can be assigned roles, data access definitions, and tracked for performance within ASOR.78

Addendum: MCP for Actions vs. Stable Integration Options

Workday's Agent Gateway supports both MCP (Model Context Protocol) and A2A (Agent-to-Agent) for third-party agent connectivity. But MCP is not the only way to wire actions into an enterprise agent platform—and it carries trade-offs that matter at scale. This section weighs MCP against more traditional integration paths.

  1. Dynamic tool discovery — An MCP server advertises its capabilities at runtime. Agents can query “what tools do you offer?” and receive a machine-readable manifest, adapting to new capabilities without redeployment.
  2. Session-aware context — Unlike stateless REST calls, MCP maintains session state across tool invocations. The agent doesn’t re-authenticate or re-establish context between steps, reducing latency in multi-step workflows.
  3. Write-once, use everywhere — A single MCP server can be consumed by IDE plugins, browser extensions, chat interfaces, and agent orchestrators alike. One integration point serves multiple surfaces.
  4. Standardization momentum — OpenAI adopted MCP in March 2025 and deprecated its proprietary Assistants API (mid-2026 sunset). Microsoft invested in the standard. The Linux Foundation’s Agentic AI Foundation now stewards governance.
  5. Structured auditability — Every tool call and response is a structured message, making it straightforward to log, inspect, and gate sensitive actions through policy layers.
  1. Security surface is large and growing — Research shows deploying just 10 MCP plugins creates a 92% probability of exploitation; even a single plugin carries ~9% risk. In early 2026 alone, 30 CVEs were filed in 60 days against MCP tooling. CVE-2025-6514 (CVSS 9.6) affected the widely-used mcp-remote package, enabling arbitrary OS command execution.
    Sources: VentureBeat, Practical DevSecOps, OWASP MCP Top 10
  2. Tool poisoning & prompt injection — Invariant Labs demonstrated that malicious tool definitions can pass seamlessly into agent contexts. Sub-techniques include “rug pulls” (malicious updates to trusted tools), schema poisoning, and tool shadowing (fake duplicates). Real incidents include the Supabase/Cursor agent exploit (mid-2025) and the Asana MCP data leak (Jun 2025).
    Sources: Data Science Dojo, Composio, AuthZed breach timeline
  3. Evolving specification — MCP is still maturing with periodic breaking changes. Early adopters carry migration risk. The spec lacks built-in support for multi-turn agent negotiation, long-running async tasks, and cross-server memory.
    Sources: Codecademy, Xenoss enterprise analysis
  4. Scalability unknowns — MCP’s session-based communication model introduces challenges at enterprise scale. Dynamic tool manifests can overwhelm agents when dozens of servers each expose many tools, creating selection ambiguity.
  5. Implementation overhead — Building a production MCP server requires schema design, resource modeling, streaming support, and auth flows—significantly more effort than exposing a REST endpoint or using native connectors.
ApproachStrengthsTrade-offs
Native REST / SOAP APIs Battle-tested, deterministic, well-documented. Existing auth (OAuth 2.0, API keys) and rate-limiting infrastructure. Decades of enterprise hardening. Stateless—agents must manage context externally. Requires per-service integration code. No dynamic discovery.
Workday Integration Cloud (EIBs, Studio) First-party support with built-in error handling, retry logic, and Workday security model. No third-party protocol dependency. Limited to Workday ecosystem. Not designed for real-time agentic tool calls. Batch-oriented.
iPaaS / Middleware (MuleSoft, Workato, Boomi) Pre-built connectors to hundreds of SaaS apps. Visual workflow builders. Enterprise governance and monitoring baked in. Licensing costs. Adds a hop. May not expose tool manifests in a format agents can dynamically consume.
Google A2A Protocol Designed for peer agent collaboration with async workflows, structured negotiation, and decentralized identity. Better fit for multi-agent orchestration. Newer than MCP. Smaller ecosystem. Workday has announced A2A support but details are thin.
IBM Agent Communication Protocol (ACP) Built-in cross-agent memory, human-in-the-loop support, long-running task management. Addresses gaps MCP doesn’t cover. IBM-originated, smaller adoption footprint. Not yet supported by Workday’s Agent Gateway.
Direct SDK / gRPC Maximum performance and type safety. Ideal for high-throughput, latency-sensitive integrations between known systems. Tight coupling. No dynamic discovery. Requires code changes for new capabilities.

For enterprise deployments of the Sana Self-Service Agent, the emerging best practice is not to choose one protocol exclusively but to layer them by use case:

  1. MCP for agent-facing tool discovery — Where agents need to dynamically find and invoke capabilities across services, MCP’s standardized manifests are the right fit. Route this traffic through an MCP Gateway (e.g., Cloudflare, Apigee) for centralized auth, rate limiting, and injection defense.
  2. REST/SOAP for deterministic back-end actions — For well-defined operations (payroll runs, benefits enrollment, compliance reports), traditional APIs remain more predictable, auditable, and hardened. Wrap them behind MCP servers if agent discovery is needed, but keep the execution path through proven infrastructure.
  3. A2A for cross-platform agent orchestration — When Workday agents need to collaborate with Salesforce, ServiceNow, or custom agents, A2A’s peer negotiation model is more appropriate than MCP’s client-server pattern.
  4. iPaaS as the governance backbone — Regardless of protocol, enterprise middleware can provide the monitoring, retry logic, and compliance controls that raw protocol connections lack.

Bottom line: MCP is the most promising protocol for agent-tool interoperability, but its security posture is not yet enterprise-grade. Organizations should adopt it with gateway controls, limit the blast radius of any single MCP server, and maintain stable API paths for mission-critical actions until the ecosystem matures.
